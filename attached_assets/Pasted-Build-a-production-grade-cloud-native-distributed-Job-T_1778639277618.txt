Build a production-grade cloud-native distributed Job Tracking Platform.

Goal:
Create a full-stack microservices-based job application tracking platform that I can run locally with Docker Compose, test with automated test cases, and later deploy to AWS with CI/CD.

Tech Stack:
Frontend:
- React.js
- TypeScript
- Vite
- Tailwind CSS
- React Router
- React Query or TanStack Query
- Axios

Backend:
- Python
- FastAPI
- SQLAlchemy
- Alembic
- Pydantic
- PostgreSQL
- Redis
- gRPC with Protocol Buffers
- SQS-compatible message queue locally using LocalStack or simple Redis queue fallback

Architecture:
Use microservices. Each backend service must be in its own folder and Docker container.

Services to build:
1. API Gateway Service
   - Public REST API entrypoint
   - Routes requests to internal services
   - JWT authentication middleware
   - Rate limiting
   - Health check endpoint

2. Auth Service
   - User registration
   - Login
   - JWT access token generation
   - Password hashing
   - User profile endpoint

3. Job Application Service
   - Create company
   - Create job application
   - Update application status
   - Filter by status, company, date
   - Track interview stages
   - Owner-scoped access only

4. Payment Plans Service
   - Free, Pro, Premium plans
   - User subscription table
   - Upgrade/downgrade plan
   - Usage limits per plan
   - Billing history
   - Mock Stripe-style webhook endpoint

5. Notification Service
   - Consume events from queue
   - Send mock email notifications
   - Store notification logs
   - Retry failed notifications
   - Dead-letter queue simulation

6. Analytics Service
   - Track number of applications
   - Interview conversion rate
   - Rejection/offer stats
   - Applications per week
   - Expose analytics API

7. AI Agent Service
   - Mock AI recommendation endpoint
   - Analyze job application data
   - Return suggestions like “follow up”, “improve resume”, “high priority”
   - Keep it modular so real LLM integration can be added later

Communication:
- Public communication: REST via API Gateway
- Internal service-to-service communication: gRPC
- Async communication: queue/events
- Example events:
  - user.created
  - job.created
  - application.status_updated
  - payment.plan_updated
  - notification.requested

Database:
Use PostgreSQL.
For local development, one PostgreSQL instance is fine, but use separate schemas or table prefixes for each service:
- auth_users
- jobs_companies
- jobs_applications
- payments_plans
- payments_subscriptions
- notifications_logs
- analytics_events

Redis:
Use Redis for:
- caching
- rate limiting
- idempotency keys
- temporary background job state

Distributed Systems Features:
Implement:
- idempotency keys for POST requests
- retries for async events
- dead-letter queue simulation
- health endpoints
- readiness endpoints
- structured JSON logging
- request correlation IDs
- rate limiting
- graceful error handling
- service-level separation
- environment-based config
